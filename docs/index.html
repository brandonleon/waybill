Waybill: regional market intel for profitable hauling runs.

Waybill is a Python CLI that scans public ESI market orders, filters to your route, and ranks cargo by net profit per cubic meter. Keep your hangar full and your jumps efficient.

No auth required Regional market data SQLite caching

Quick Start

uv run waybill route \\ --from-station "Hek VIII - Moon 12 - Boundless Creation Factory" \\ --to-system "Aldik" \\ --cargo-m3 60000 \\ --min-profit 500000 \\ --instant-only

Pulls regional orders, filters your route, returns top ISK/m³ runs.

Overview

Waybill pulls regional market orders, narrows them to your stations or systems, computes net profit after broker and tax fees, and ranks opportunities by ISK per cubic meter. It stores responses in SQLite so you can iterate quickly without hammering ESI.

Profit ranking: ISK/m³ Detour hints supported Public ESI endpoints

Requirements

  • Python 3.12+
  • uv for virtualenv and dependency management
  • Network access to ESI public endpoints

Designed for capsuleers who want fast regional scans without authentication overhead or structure data.

Getting Started

Waybill installs as a CLI entrypoint. Use uv to create a virtual environment and install in editable mode.

uv venv uv pip install -e .
uv run waybill --help

Commands

route

Find profitable hauling opportunities between systems or stations.

uv run waybill route \\ --from-system "Hek" \\ --to-system "Aldik" \\ --cargo-m3 60000 \\ --min-profit 100000 \\ --instant-only
  • Ranks by ISK/m³ then liquidity.
  • Supports off-route detour hints.

best-sell

Scan the best stations to sell a specific item within a jump radius.

uv run waybill best-sell \\ --item "Tritanium" \\ --from-system "Aldik" \\ --max-jumps 5 \\ --sec-band hi \\ --instant-only
  • Supports security filters or sec-band shortcuts.
  • Outputs ISK/m³ and net price.

sync-items

Refresh cached item metadata for known type IDs.

uv run waybill sync-items --verbose
  • Repeat --region to target specific regions.
  • Use --refresh to override cache age.
  • --limit refreshes the oldest cached types first.

Option Highlights

  • --from-station / --from-system
  • --to-station / --to-system
  • --cargo-m3 and --min-profit
  • --instant-only for quick flips
  • --broker-fee and --sales-tax
  • --off-route-jumps for detour scans
  • --market-cache-ttl and --refresh-market
  • --page-workers for faster region pulls

Security Filters

  • Security status ranges from -1.0 (deep null) to 1.0 (high-sec).
  • Use --min-security / --max-security for precision.
  • Or use --sec-band hi|low|null|all as shorthand.
  • --avoid-low is an alias for --min-security 0.5.

Config & Cache

Waybill uses a local SQLite cache at ~/.cache/waybill/waybill.sqlite. Cache TTLs and item refresh behavior can be tuned with an optional config file.

  • Default market cache TTL: 14,400 seconds (4 hours).
  • Cache status is printed even without --verbose.
  • Item type refresh runs after route and best-sell.
~/.config/waybill/config.toml [item_type_refresh] oldest_limit = 10 max_age_seconds = 0 [market_cache] ttl_seconds = 14400

Output Columns

route

  • Item, Buy, Sell, Net Profit, ISK/m³, Qty, Total Volume
  • Mode column appears when non-instant runs are included

best-sell

  • Station, System, Region, Jumps, Sec
  • Net Price, ISK/m³, Volume, Mode

Off-route Table

When you set --off-route-jumps, Waybill prints an additional table that compares base net profit against detour opportunities.

  • Delta shows the extra profit from the detour.
  • Detour indicates jumps off the shortest path.
  • Station and system context are included for quick scouting.

FAQ

Does Waybill use authentication?

No. Waybill uses public ESI endpoints, so it cannot see player structure data.

Why does it fetch entire regions?

ESI market data is regional. Waybill pulls full regions then filters to your route so results are accurate.

How do I speed up scans?

Increase --page-workers or reuse cached data with a higher TTL.

Limitations

  • No player structure data without authentication.
  • Market pulls can be large; caching is recommended.
  • Item type cache is lazy unless sync-items is run.

Performance Tips

  • Set --market-cache-ttl 14400 or higher for repeated runs.
  • Use --page-workers 2 or 4 for large regions.
  • Keep --verbose enabled for page-level progress.
